# P0-10 会话线程丢失后自动恢复

## 用户故事

作为培训负责人，当网络波动、服务重启等原因导致对话线程失效时，我希望系统能自动重建会话，不需要手动刷新页面或重新创建工作区，历史对话记录仍然可见。

## 场景描述

LangGraph 线程因 404 或过期失效时，前端 Assistant 组件检测到错误，自动将 threadId 置空，useStream hook 自动新建线程，新线程 ID 持久化回后端，用户几乎无感知地恢复到可正常对话的状态。

## 前置条件

- 用户已在工作区进行过对话，存在 thread_id 记录
- LangGraph 服务重启或线程因某种原因失效

## 核心步骤

1. 用户发送消息，stream 返回 404 或 "Thread not found" 错误
2. 前端 `stream.error` 捕获到含 "404"/"not found"/"Thread" 的错误
3. Assistant 组件自动将 `threadId` 设为 `null`
4. `useStream` hook 检测到 `threadId=null`，自动创建新线程
5. 新的 `threadId` 通过 `updateWorkspaceThreadId` 持久化到后端
6. 用户可继续发送消息，对话正常恢复

## 验收标准

- [ ] 线程 404 后，5 秒内自动恢复可发送状态，无需用户手动操作
- [ ] 新 threadId 成功写回后端，下次进入工作区仍使用新线程
- [ ] 恢复期间聊天输入框保持禁用（isLoading），恢复后自动解禁
- [ ] 非线程相关的错误（如网络超时）不触发线程重建逻辑
- [ ] 错误信息在对话流中以友好文字显示，不暴露技术错误码

## 异常场景

- resume 场景下线程丢失：重建线程后，interrupt 表单不再重复展示（hasResumed 检测）
- 连续多次线程失效时，每次均能正常触发重建，不进入死循环
